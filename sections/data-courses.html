Clicking + Add New Course (or editing an existing course) opens a detailed setup screen. This screen defines how Edario is allowed to schedule the course.

3.1 — Identity fields
  • Course Code (text field)
  • Course Name (text field)

These should be consistent and unique enough to avoid collisions after upload. Clean naming also makes reports and search far easier later.

Recommended workflow:

  • Use codes consistent with your SIS/course catalog.
  • Keep names recognizable (avoid abbreviations that confuse staff).
Recommended GIF/video: “Add a new course with code + name.” Replace data-courses-add-course-code-name.mp4 with your real filename.
3.2 — Elective classification

The Elective section is typically a dropdown (example shown: Non-elective). This affects how the system thinks about requests and balancing.

  • Non-elective: core courses often have stricter expectations
  • Elective: often requires broader balancing and demand-aware placement

Set this accurately so later reports reflect real scheduling priorities.

Recommended GIF/video: “Switch Non-elective → Elective.” Replace data-courses-elective-toggle.mp4 with your real filename.
3.3 — Semester / term length

A Semester dropdown defines how long the course runs (example shown: FY – Full Year).

Common meanings:

  • FY = Full Year
  • S / S1 / S2 = Semester patterns
  • Q / Q1 / Q2 = Quarter patterns (if applicable)

This term definition influences scheduling feasibility and alignment with student requests.

Recommended GIF/video: “Set course semester type.” Replace data-courses-set-semester-type.mp4 with your real filename.
3.4 — Period span (periods per section)

A critical control: “How many periods does a section of this course span?” with buttons 1, 2, 3, 4, 5, 6.

This defines whether the course is:

  • Single-period
  • Double-block
  • Longer multi-period block

Edario uses this later in placement to ensure the section occupies the correct number of consecutive time blocks.

Scheduling impact: If span is wrong, placements may look “impossible” or create artificial conflicts.
Recommended GIF/video: “Change span from 1 → 2 periods.” Replace data-courses-change-span.mp4 with your real filename.
3.5 — Credits

A numeric Credits field exists (example shown: 0).

Credits are commonly used for:

  • Reporting
  • Transcript mapping
  • Policy/requirements mapping (varies by school)

Even if credits don’t change placement logic directly, keep them accurate for downstream reporting.

Recommended GIF/video: “Set credits and explain what it means.” Replace data-courses-set-credits.mp4 with your real filename.
3.6 — Priority (1–100)

A numeric Priority field (range 1–100) controls scheduling preference.

  • 1 = highest priority
  • 100 = lowest priority

This is a scheduling-control lever: higher priority courses tend to be scheduled with more protection when conflicts exist.

Use case: Raise priority for required/core courses and time-sensitive offerings. Leave lower priority for flexible electives where conflicts are more tolerable.

Course constraints Room restrictions, availability, caps, requests, special flags
In Progress

The course edit screen includes multiple constraint sections that control where and how the course can be scheduled: classroom restrictions, availability grids, section caps, class size, student requests, and special options.

4.1 — Classroom restrictions (hard requirement)

Section title:

“Classroom(s) this Course NEEDS to be in”

  • Empty state may show: No classrooms set yet!
  • Action: Add Classroom Restriction

This is a hard constraint: the course can only be scheduled into those classrooms.

Recommended GIF/video: “Add a Classroom Restriction to a course.” Replace data-courses-add-classroom-restriction.mp4 with your real filename.
4.2 — Course Availability (period checklist + day grid)

Section title: Course Availability

A) Period Availability checklist (left)

  • Vertical list of periods (Period 1, Period 2, Period 3, Period 4/5, Period 5/6, Period 6/7, Period 7/8, Period 9, Period 10…)
  • Check/uncheck periods to allow/disallow course in those periods.

B) Day-by-day availability matrix (right)

  • Grid columns: Day 1 … Day 5
  • Grid rows: each period
  • Cells indicate Available (green) or Unavailable (red)
  • Cells can be toggled to encode day/period specific constraints.

Common use cases:

  • “This course can never run Period 1.”
  • “This course is unavailable on Day 3 Period 6/7.”
  • “Only run on certain days/times.”

Availability becomes extremely powerful when combined with teacher availability and classroom restrictions later.

Real example: labs blocked first period to allow setup time, or a shared resource that is only free on certain days.
Recommended GIF/video sequence: “Uncheck a period on the left” + “Toggle a red/green cell” + “Explain a real example.” Replace data-courses-availability-grid.mp4 with your real filename.
4.3 — Max Number of Sections

A dropdown labeled Max Number of Sections (example shown: None).

  • None means no cap.
  • A number caps how many sections Edario can create/consider for this course.

Use cases:

  • Limited staffing
  • Program constraints (e.g., only 2 sections offered no matter demand)
  • Controlled pilots for new electives
Recommended GIF/video: “Set max sections for a popular course.” Replace data-courses-max-sections.mp4 with your real filename.
4.4 — Max Class Size

A dropdown labeled Max Class Size (example shown: 30).

This affects:

  • Section balancing
  • Seat availability
  • Later student placement decisions

Use cases:

  • Small specialty classes (cap students)
  • Safety constraints for labs/shops
  • Room capacity or staffing limitations
Tip: If you cap class size aggressively, watch Schedule Summary to confirm matched requests don’t fall unexpectedly.
Recommended GIF/video: “Lower max class size and show how it limits seats.” Replace data-courses-max-class-size.mp4 with your real filename.
4.5 — Students Who Have Requested This Course

Section title: Students Who Have Requested This Course

  • Shows empty-state message if none exist.
  • Includes button: Add Student Request
  • Shows Total Requests count

This ties Courses directly into Requests (Scheduling Data). It’s a useful place to validate demand or test behavior.

Recommended GIF/video: “Add a student request to a course.” Replace data-courses-add-student-request.mp4 with your real filename.
4.6 — Other Course Options (special flags)

Section title: Other Course Options

These Yes/No toggles affect later scheduling logic:

  • Is Lunch? — Yes if this course is a lunch course.
  • Is Inclusion? — Yes if this is an inclusion course (special ed combined with general ed).
  • Does this course have an extra lab section? — lab once a week on top of daily class.
  • Can a student swap this class for his/her lab? — allows missing class to attend lab.
  • Should this course be included in the schedule? — choose No if offered outside official classroom (online/community college).

Why these matter later:

  • Section creation logic
  • Lab balancing
  • Inclusion pairing/subsections
  • Avoiding scheduling “fake” courses into the grid
Tip: If you mark a course as “Not in schedule / not included,” confirm it disappears from placement boards and balancing reports where expected.
